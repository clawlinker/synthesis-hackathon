# SVG Generation Research for Agent Receipts

## Objective
Select the most efficient and verifiable way to generate SVG receipt cards for autonomous agents.

## Candidates

### 1. Satori (@vercel/og)
- **Pros:** Converts HTML/CSS (JSX) to SVG. Supports Tailwind (subset). High-fidelity, very modern.
- **Cons:** Tailwind v4 support might be tricky (Satori uses an older engine). Heavy dependency tree.
- **Verdict:** Best for "Social Card" style high-fidelity receipts.

### 2. Manual SVG Template (JSX/TS)
- **Pros:** Zero dependencies. Extremely fast. Full control over verifiable data embedding.
- **Cons:** Harder to design complex layouts (no flexbox/grid without manual math).
- **Verdict:** Best for "Bare Metal" verifiable receipts that need to be generated by constrained agents.

### 3. Sharp
- **Pros:** Industrial grade. Can convert SVG to PNG/WebP easily.
- **Cons:** Native dependency (can be finicky in some environments). No HTML->SVG path.
- **Verdict:** Useful if we need raster output, but not for the SVG generation itself.

## Decision: Hybrid Approach
1. **Design in React/Tailwind v4** for the web UI.
2. **Implement a "Lite" SVG Template** using standard JSX (without heavy libs) for the actual receipt generation to keep it verifiable and lightweight.
3. If fidelity is a blocker, adopt **Satori** for the server-side generation.

## Edge Cases
1. **Dynamic Text Overflow:** Long ENS names or long skill names must be truncated correctly in SVG (use `text-overflow: ellipsis` equivalent logic).
2. **Dark/Light Mode:** Receipts should probably have a fixed "Theme" (e.g., Cyberpunk/Dark) so they look consistent when shared on social media.
3. **Verification Hash:** Embed the transaction hash or a signature as a `<metadata>` or `data-*` attribute in the SVG so the file itself is a self-contained proof.

## Verified Edge Case
- **Long Addresses:** Tested SVG truncation logic. Manual math for `viewBox` is required if using raw templates.
